35.2 计算频率

学习如何有效地编程GPU的下一步，是学会利用GPU由多个不同处理器组成的这一事实。考虑典型的栅格化管道。在硬件实现中，该管道由一系列以并行方式运行且具有不同能力、可编程性程度、优势和劣势的处理器组成。在管道的每个阶段之间有一个工作队列。只要一个管道阶段的输出工作队列未满，该阶段就可以与其他阶段并行工作。当CPU忙于将几何体和状态信息传递给GPU时，GPU的顶点处理器可以处理到达的每个顶点，对其进行变换等操作。同时，栅格化器可以将一组经过变换的顶点转换为片段（可能产生大量片段），并将这些片段排队等待片元处理器进行处理。 注意到管线中每个阶段的工作量通常是递增的：几个顶点可能会生成许多片段，而每个片段的处理成本可能都很高。鉴于每个阶段工作量的这种相对增加，从概念上将这些阶段视为一系列嵌套循环（尽管每个循环都与其他循环并行运行，如前所述）是有帮助的。这些概念上的嵌套循环在伪代码中所示，参见列表35-1。

例 35-1。标准光栅化管道作为嵌套循环系列

对于每次执行的操作，我们必须考虑到该操作的计算开销有多大，以及它被执行的频率有多高。在普通的CPU程序中，这一点相对容易理解。与这里仅仅概念性展示的嵌套循环不同，当实际面对一系列嵌套循环时，我们不难发现：位于内层循环中的某个表达式可能是与循环无关的（loop-invariant），也就是说，它可以被“提升”到外层循环中，并以较低的频率进行计算。类似地，在CPU程序中通常会避免在内层循环中出现分支语句（branch），因为分支操作本身是相当耗时的；而如果它出现在内层循环中，那么由于内层循环可能被执行成千上万次，这个分支操作也会相应地被执行多次，从而显著增加整体的计算开销。 在编写GPU程序时，特别需要注意减少冗余工作的量。很自然地，之前讨论的那些用于降低CPU程序计算频率的技术同样适用于GPU程序。但是由于GPU编程的本质特性，列表35-1中的每一个概念上的嵌套循环实际上都是一个独立的程序，在不同的硬件上运行，并且可能使用了完全不同的编程语言编写。这种分离使得我们很容易忽略这些类型的优化。

The translated section title in the context of "Precomputation of Loop Invariants" is: **35.2.1 预先计算循环不变量**

新入行的GPU程序员最常犯的第一个错误是，在片段程序中无谓地重新计算那些在几何原Primitive内线性变化或均匀分布的值。纹理坐标就是一个典型的例子。它们在线程被绘制的区域上是线性变化的，而光栅化器会自动进行插值处理。然而，当使用多个相关的纹理坐标（例如第35.1.1节示例中的偏移和邻近坐标）时，一个常见的错误是在片段程序中计算这些相关值。这会导致可能昂贵的计算过程被频繁执行。 “将相关的纹素坐标计算移到顶点程序中进行，这样做会好得多。虽然这实际上只是转移了负载，并且光栅化器中的插值仍然是按片处理的操作，但问题在于每个管道阶段的工作量有多大，以及这些工作需要执行的频率有多高。无论是哪种方式，最终结果都是一组在绘制的基本体上呈现线性变化的纹素坐标。然而，插值通常比在每个片段的基础上重新计算给定值要便宜得多。只要片段的数量远远超过顶点数量，将大部分计算转移到按顶点而不是按片段进行处理是有意义的。” 然而，值得再次强调的是，无论最终是否用于索引纹理，任何在域内线性变化的值都可以通过这种方式进行计算。这是理解GPU编程的关键之一：只要了解"专用"GPU功能对应于通用概念的方式，它们的名称基本上是不相关的。请查阅本书第31章《计算概念到GPU的映射》，深入了解这些对应关系。 为了更进一步地将循环不变量代码提升到GPU之外，有些值最好在CPU上预先计算，而不是在GPU上。任何在整个绘制几何体过程中保持恒定的值都可以完全提取到CPU，并作为统一参数传递给GPU程序。这有时会导致语义上不太优雅的参数；例如，假设我们将一个统一参数“size”传递给顶点或片段程序，但该程序仅使用“size * size * 100”的值。尽管“size”甚至“size squared”在语义上有意义，但“size squared times 100”几乎没有任何语义上的含义。然而，如果我们将GPU程序视为一系列嵌套循环，并且目标是消除重复的循环不变量计算，那么很可能仍然更倾向于在CPU上计算“size * size * 100”，并将该值作为统一参数传递给GPU。

35.2.2 使用查找表进行预计算

从更传统意义上讲，“预计算”是指离线完成的预先计算——这是经典的存储与计算之间的权衡。这一概念也可以很好地映射到GPU上：函数具有固定大小的定义域和值域，并且在整个算法运行过程中保持不变——即使它们在复杂的方式下根据输入发生变化，这些函数也可以被预计算并存储在纹理贴图中。 “纹理贴图可用于将一维、二维或三维函数存储在有限域中的1D、2D或3D纹理中。通常，纹理通过0到1之间的浮点纹理坐标进行索引。该范围由所用的纹理格式决定；8位纹理格式只能存储[0, 1]范围内的值，但浮点纹理提供了更大的可能值范围。纹理最多可以存储四个通道，因此您可以在同一个纹理中编码多达四个独立函数。纹理查找还提供过滤（插值），您可以利用它来获取表项之间值的分段线性近似。” 举个例子，假设我们有一个片段程序，想将其应用于棋盘图案：一个大四边形（quad）的一半片段，“红色”的那些，在第一遍处理中被处理，而另一半，“黑色”片段，则在第二遍处理中进行处理。但是，当前正在处理的片段是红色还是黑色，这个片段程序如何判断呢？一种方法就是对片段的位置使用取模运算： 
// Calculate red-black (odd-even) masks
float2 intpart;
float2 place = floor(1.0f - modf(round(position + 0.5f) / 2.0f, intpart));
float2 mask = float2((1.0f - place.x) * (1.0f - place.y), place.x *place.y);
if (((mask.x + mask.y) && do_red) || (!(mask.x + mask.y) && !do_red))
{
  // ...
}
大致来说，这里的“place”指的是片段位置对2取模的结果，而mask.x和mask.y是两个布尔值，结合起来可以告诉我们这个片段是红色还是黑色。统一参数do_red被设置为0时，表示我们要处理黑色的片段；设置为1时，则表示我们要处理红色的片段。 显然，为了这个看似简单的任务，做这么多工作是荒谬的。预先计算一个棋盘格纹理会简单得多，该纹理在黑色texel中存储0，在红色texel中存储1。然后我们可以跳过所有前面的算术运算，用一次纹理查找来代替，从而实现显著加速。剩下的就是以下内容：

然而，尽管在这种情况下使用查找表在性能上是一个优势，但并不总是如此。许多基于GPU的应用程序——尤其是那些大量使用四分量浮点纹理采样的应用程序——是受内存带宽限制的。因此，为了节省少量计算而引入额外的纹理读取可能会带来损失而不是收益。此外，纹理缓存一致性至关重要；如果查找表以不一致的方式访问，将会导致缓存颠簸，从而损害性能而非提升。然而，如果可以将足够的计算“预烘焙”（pre-baked），或者如果相关GPU程序已经是计算受限的，并且从纹理中读取烘焙后的结果是以空间一致的方式进行，那么使用查找表可以显著提高性能。有关基准测试和GPU应用程序剖析的更多信息，请参阅本章后面的第35.3节。

35.2.3 避免内循环分支

在CPU编程中，通常希望避免在内循环内部出现分支。这种做法通常涉及创建多个循环副本，每个副本处理数据的一个子集，并遵循特定于该子集的执行路径。这种技术有时被称为静态分支解析或子流处理。 同样的概念也适用于GPU。因为片段程序从概念上讲代表一个内部循环，所以应用这一技术需要将包含分支的片段程序分解为多个没有分支的片段程序。每个结果程序对应原始单一庞大程序中的一条代码执行路径。此技术还要求应用程序细分数据，对于GPU来说，这意味着要对多个原语进行光栅化而不是一个原语。有关详细信息，请参阅本书第34章“GPU流程控制习惯用法”。 一个典型的例子是一个2D网格，其中位于网格边界的数据元素需要特殊的处理，而内部数据元素则不需要。在这种情况下，最好创建两个独立的片段程序：一个是简短的、不考虑边界条件的，另一个是较长的、考虑边界条件的，并且分别在内部区域绘制实心四边形，在边界区域绘制轮廓四边形。

The best translation of "35.2.4 The Swizzle Operator" into Chinese, considering its context in computer graphics, is: **35.2.4 Swizzle 操作符** This translation ensures clarity and consistency with existing technical literature.

一个容易被忽视或未充分利用的GPU编程特性是“swizzle操作符”。由于GPU上的所有寄存器都是四分量向量，但并非所有指令都接受四分量向量作为参数，因此需要一种机制来从这些四分量寄存器中创建其他大小的向量。Swizzle操作符提供了这一功能。它在语法上类似于C语言中的结构体成员访问，但其有趣之处在于可以任意重新排列、复制或省略数据成员，如以下示例所示： 
float4 f = float4(1.0f, 2.0f, 3.0f, 4.0f);
float4 g = float4(5.0f, 6.0f, 7.0f, 8.0f);
float4 h = float4(0.1f, 0.2f, 0.3f, 0.4f);
// note that h.w is syntactically equivalent to h.wwww
float4 out = f.xyyz * g.wyxz + h.w; // this is one instruction!
                                    // multiply-and-accumulate again.
// result: out is (8.4f, 12.4f, 10.4f, 21.4f)

“35.3 剖析与负载均衡”

就像CPU编程一样，编写高效GPU代码的最佳方法也是先写出能正常工作的简单代码，然后再根据需要逐步优化。遗憾的是，目前这个过程在GPU代码中并不那么轻松，因为缺少帮助它的工具。不过，只要程序员多加一点用心，即使没有特殊工具，也能有效地进行优化。 在图形编程中，频繁的时间测量至关重要。实际上，每一个应用的优化都应该通过计时来独立验证。即使那些看似明显且必然带来速度提升的优化，也可能完全没有效果，甚至导致性能下降。这种情况出现时，可能表明着色流水线中的负载不平衡问题。然而，这类问题还可能源于图形驱动程序内部进行的最终优化步骤，这使得程序员难以确切了解实际执行的代码是什么。对源代码进行看似有益的转换，实际上可能会禁用驱动程序正在执行的其他某些优化，从而导致整体性能下降。通过在每次优化后进行基准测试，这些类型的问题会更容易被检测到；事后回溯以隔离有问题的优化，只会浪费时间和精力。 一旦显而易见的优化措施被应用之后，对硬件能力的低级理解变得有用，特别是在超标量指令执行方面。你对能够同时执行的指令类型了解得越详细，就越有可能识别出代码转换和改进的额外机会。NVShaderPerf工具通过向您展示您的片段程序如何调度到GPU的算术单元上，从而帮助您实现这一点，大大减少了片段程序优化中的猜测工作。（有关NVShaderPerf的更多信息，请参见本章末尾的“参考资料”。） 即使是最优化的片段程序，仍然有可能让 GPU 应用程序运行得更快。此时，关键在于找到应用程序的主要瓶颈，并尽可能地将负载转移出去。第一步是使用 CPU 应用程序剖析工具，如 Rational Quantify、Intel VTune 或 AMD CodeAnalyst。这些工具有助于识别问题，例如驱动开销（如上下文切换）。需要注意的是，并非Profiler 提供的所有信息都是可靠的；许多图形 API 调用是非阻塞的，以便 CPU 和 GPU 可以并行运行。在某些情况下，GPU 的工作队列会填满，CPU 侧的一个任意 API 调用会被阻止，等待 GPU 追上。被阻止的调用将被 CPU Profiler 计为执行时间很长，即使在此期间完成的工作更公平地应归因于之前未被阻止的 API 调用。只要牢记这一点，CPU 嘞profilers 仍然可以提供有价值的见解。 一旦CPU的开销被最小化，剩下的唯一问题就是确定GPU瓶颈出现在哪里。为了这个目的，市场上开始出现专门用于监控GPU性能的工具；NVPerfHUD就是一个很好的例子（更多关于NVPerfHUD的信息请参见“参考资料”）。在这些工具尚未提供详细信息或不适用于特定应用程序的情况下（例如，NVPerfHUD目前仅限于Direct3D，并且不适合GPGPU应用），可以利用其他技术来填补空白。每种方法都涉及一系列实验。一种方法是测试将工作量添加到GPU处理管线的某个部分是否会增加总执行时间，或者减少工作量是否会缩短执行时间。如果是这样，那么很可能会发现瓶颈就出现在那里。例如，如果一个片段程序是计算受限的（compute-limited），那么插入额外的计算指令会增加程序的执行时间；但如果该程序是内存带宽受限的（memory-bandwidth-limited），则可能可以免费完成这些额外的计算。另一种方法是对图形卡的计算核心或内存系统进行降频测试（underclock）。如果内存系统可以在不降低性能的情况下降频，那么内存带宽就不是瓶颈因素。这些技术和更多内容在NVIDIA GPU编程指南中有详细说明（NVIDIA 2004）。 “最终，理解瓶颈是消除它们的关键。前面提到的技术和一点经验有助于理解这一点。”

35.4 结论

正如我们所见，GPU程序优化与CPU程序优化有许多共同的主题。为GPGPU提供统一流编程模型的高级语言正在兴起（McCool等人于2002年、Buck等人于2004年），这有望带来更广泛的全局GPU应用程序优化机会，同时GPU代码的自动优化技术也在不断提升。然而，在这些技术成熟之前，GPU程序员仍需承担起高层次优化的责任，并深入了解他们所使用的硬件细节，以充分利用低层次优化的优势。希望本章能为您提供所需的工具，助您掌握如何最大限度地从您的GPU应用中挖掘出GPU所能提供的全部性能。
